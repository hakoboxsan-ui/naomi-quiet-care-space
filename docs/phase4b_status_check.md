# Phase 4-B 実装ステータス確認レポート

本ドキュメントは、NAOMIプロジェクトにおける **Phase 4-B（低圧応対と安全性、アクセシビリティの統合）** の実装状況について、現時点のソースコードおよび構成を分析・確認した結果をまとめたものです。

---

## 1. 確認項目と判定

| 確認項目 | 実装の有無 | 詳細・実装場所 |
| :--- | :---: | :--- |
| **PersonaGuard 追加** | **有** | `agent/persona_guard.py` に非診断・受容的表現への置換フィルターが実装されています。 |
| **EmotionEngine 接し方制御** | **有** | `agent/emotion_engine.py` にユーザーの心身状態に応じた返答スタイル（短文化、返事強要の削減、睡眠誘導など）の自動調整ロジックが実装されています。 |

## 2. 変更・追加済みファイル一覧

本フェーズ（Phase 4-B）の実装に関連して、追加・変更された主要ファイルは以下の通りです。

1.  **`agent/persona_guard.py` (新規追加)**
    *   医療的断定（「治療」「診断」「病気」）や命令・説教表現（「すべき」「しなさい」）を、非診断的かつ受容的な表現（「ケア」「整理」「してみてくださいね」）へ自動マッピング・正規表現置換するガードレールモジュール。
2.  **`agent/emotion_engine.py` (新規追加)**
    *   ユーザーの `stress`（高い場合は強制的な短文化とアドバイスのカット）、`energy`（低い場合は返答要求の免除文を付与）、`sleepiness`（高い場合は目を閉じて休むよう促す最小テキストに差し替え）、`reassurance_need`（高い場合は受容・安心表現の先頭付与）をコントロールする感情適応エンジン。
3.  **`agent/core.py` (変更)**
    *   アクセシビリティタブでのボタン入力（つらい、不安、息苦しい等）を受け取り、仮の状態推定・会話圧・サマリー・スタッフノートを瞬時に割り当てる `process_accessibility_input` メソッドを追加。
    *   応答文、スタッフノート、ケア提案、整理サマリーの生成後に `EmotionEngine` と `PersonaGuard` を通して事後処理（`_post_process_response`）を適用するパイプラインを構築。
4.  **`frontend/streamlit_app.py` (変更)**
    *   「大きな文字モード」を有効にするチェックボックスと、フォントやボタンサイズを最適化するカスタムCSSトグルを追加。
    *   タップだけで状態（つらい、痛い、息苦しいなど）を伝えるボタン群と、AIの応答方法（文字、ボタン、短く、話したくないなど）を制御するボタン群の実装。
    *   ボタン入力時にバックエンドの `process_accessibility_input` を発火させる処理の統合。
    *   医療診断を行わない旨を明記した「安全のためのご案内（免責枠）」を新設。
5.  **`docs/voice_accessibility_plan.md` (新規追加)**
    *   高齢者・視覚障害者のための音声デザイン（早口の厳禁、短文、低圧、非干渉）と将来のTTS（Style-BERT-VITS2/Noaモデル、irodori_TTS）連携のロードマップ。
6.  **`docs/camera_future_plan.md` (新規追加)**
    *   将来的なカメラ画像による表情分析（FACSアクションユニット検出）と状態推定を用いた、能動的チェックインのシステム構成メモ。

## 3. 未完了箇所・途中停止箇所

### 未完了箇所（今後の拡張余地）
*   **音声合成（TTS）の実装連携**
    *   `docs/voice_accessibility_plan.md` として設計方針は定められていますが、実際のStyle-BERT-VITS2等のサーバー接続や音声再生のコードは組み込まれていません。
*   **音声入力（STT）の連携**
    *   話しづらい人向けの音声による文字入力インターフェースは未搭載です。
*   **スクリーンリーダー対応の完全最適化**
    *   HTMLのセマンティクス（WAI-ARIA等）を厳格に割り当てるフロントエンド側のチューニングは行われていません。
*   **カメラ表情解析の統合**
    *   `docs/camera_future_plan.md` のカメラ連携構想は、現状のStreamlit UI上にはモジュールとして実装されていません。

### 途中停止・保留箇所
*   **`agent/core.py` 内の「問診モードへの自動移行判定」**
    *   LLM（Gemini）によって「現在問診による状態整理が必要か」を自動検出するロジック（`detect_need_for_intake_mode`）は `GeminiBrain` に実装されています。
    *   しかし、`agent/core.py` の L141–146 付近において、この自動移行機能は意図的にパス（コメントアウト）されています。これは、デモ操作時に意図しないタイミングで自動的に問診モードへ切り替わってしまうことを防ぎ、UI側のボタンから手動で起動（コントロール）させる挙動を優先するための設計上の判断（保留）となっています。
