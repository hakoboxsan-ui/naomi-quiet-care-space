# NAOMI 現状確認レポート (Current NAOMI Overview)

本ドキュメントは、現時点における人間状態適応型傾聴AI「NAOMI」の開発進捗、UIの思想、システムの安定性、未実装項目、および今後のロードマップについて客観的に整理したレポートです。

---

## 1. 現在実装済みの主要機能

現時点で、以下の機能がコアエンジンおよび Streamlit UI 上に実装されています。

*   **Listening First (段階的受容)**
    *   `agent/emotion_engine.py` および `agent/behavior_policy.py` を通じ、ユーザーの不安や安心要求度（`reassurance_need`）が高い場合には、解決策の提示を抑え、まずは「お気持ちを受け止める」対話表現を最優先するロジックが組み込まれています。
*   **Pressure Control (会話圧制御)**
    *   `VERY_LOW`, `LOW`, `MEDIUM`, `HIGH` の4段階の会話圧（Pressure Level）を定義。
    *   ユーザーのストレス度合いが高いときやエネルギーが低いときは、自動的に返答テキストを短文化し、過度なアドバイス（箇条書きなど）を削って認知的負荷を減らす制御（会話圧の低下）が行われます。
*   **Care & Intake Demo (状態整理デモ)**
    *   `agent/intake_manager.py` が内科領域（`internal_medicine`）を模した15項目の聞き取り（発症時期、主症状、随伴症状、持病、服薬状況など）ステップを制御。
    *   UI上では進捗がプログレスバーで可視化され、最終的にお伺いした内容を客観的にまとめた「状態整理サマリー」が生成されます。
*   **Staff Note (支援者向け共有メモ)**
    *   対話から推定された状態、普段との違い、ケア提案、会話圧などを介護者や看護師、ご家族向けに客観的な文章として自動構成するレポート作成機能（`generate_staff_note`）が搭載されています。
*   **Personal Baseline (個人基準値比較)**
    *   `agent/personal_baseline.py` が利用者の「普段の元気度」「会話量」「睡眠傾向」を保持（`data/personal_profiles.json`）。
    *   現在の入力時状態とベースラインを比較し、いつもより元気がなさそう、言葉が極端に少ないなどの「普段との違い（Baseline Diff）」を検出し、ケア提案やスタッフノートに反映します。
*   **Accessibility入口 (やさしい入力)**
    *   `frontend/streamlit_app.py` の「Accessibility / やさしい入力」タブに以下を統合。
        *   **大きな文字モード**：CSSによるフォントサイズおよびボタンのサイズ変更。
        *   **簡易状態申告ボタン**：「つらい」「不安」「痛い」「息苦しい」などのボタンから、直接仮の人間状態をシミュレート。
        *   **応答制御ボタン**：「文字で答える」「返事は短く」「今は話したくない」などを選ぶことで、AIの応答トーンや長さを変えられます。
*   **Demo Package & Presentation Flow**
    *   「通常のAI（即座に箇条書きで解決策を押しつける）」と「NAOMI（状態を受け止め、会話圧を下げ、余白を作る）」を左右に並べて比較表示するエリアを完備。
    *   キーワード検出型の固定デモシナリオ（疲労、不安、孤独、限界など計9パターン）を搭載しており、APIキーが無い環境でも一貫した動作比較を実演可能です。

## 2. 現在のUI方向性

NAOMI のUIは、スマートスピーカーのような「早くて、ハキハキとした情報伝達」とは真逆の、**低圧かつ非干渉**なデザイン思想に立脚しています。

*   **Quiet Luxury (静かなる心配り)**
    *   原色を排除した落ち着いた色彩設計と、ダークなトーンを取り入れた配色。
    *   返答は極力1〜2文の短い構成に留め、ユーザーに素早い回答や文字入力を強制しない「お返事は気が向いたときで大丈夫です」といった非干渉な余白を提示します。
*   **Cognitive Relief (認知的休息) と状態整理**
    *   ユーザーのストレスが高い時には情報を意図的に引き算し、「頭を休めること」を最優先させます。
    *   ユーザー自身の心身の状態を人に話せるレベルへ「整理」する手助けをし、介護施設・医療の受付前段、あるいは家庭内での気づきレポートとして橋渡し（Handoff）をする役割を担っています。
*   **Accessibility の具現化**
    *   高齢者や視覚機能に負担がある方でも読みやすい大文字化トグル、キーボード入力が困難な状況下でもタップだけで今のつらさを伝えるボタン設計など、誰一人取り残さないためのやさしいインタラクションを指向しています。

## 3. 安定性

デモシステムとして非常に高いロバスト性（堅牢性）を誇っています。

*   **Streamlit起動安定性**
    *   軽量な設計により、環境依存によるクラッシュがなく、ローカルで即座に安定起動します。
*   **GeminiなしのFallback**
    *   APIキーが設定されていない、あるいはネットワーク障害が発生した場合でも、自動で完全なローカルフォールバックが走る設計になっています。
    *   キーワード検出（`detect_demo_scenario`）によるモック応答と、ルールベースの状態推論（`estimate_state`）、応答生成（`generate_response`）の連動により、デモの主要フローが完璧に動作します。
*   **Intakeの安定性**
    *   シンプルなステップ進行（インデックスカウンタ方式）のため、対話中の無限ループや進行不能バグが起きない安全な設計です。
*   **Red Flag & PersonaGuard 安全性**
    *   「息苦しい」「胸が痛い」などの緊急キーワードに対し、危険度「High」のレッドフラグ警告を表示する仕組みがローカルで確実に動作します。
    *   `persona_guard.py` による置換フィルターにより、AIが医療的・断定的・あるいは命令的な口調で発話しそうになった場合（「治療」「診断」「病気」「しなさい」など）を、自動的に非診断的かつ受容的な表現（「ケア」「整理」「してみてくださいね」）へ丸め込むことが保証されています。
*   **現時点の危険箇所 (注意点)**
    *   `personal_profiles.json` への書き込み例外処理：書き込み時にクラッシュした際、データが空で初期化される可能性があります（デモ用途では重大な問題にはなりにくいですが、運用時は注意が必要）。
    *   LLMからのJSONパース：Gemini接続時、返却JSONのフォーマット崩れに対する耐性は簡易的な文字列置換に留まるため、稀にフォールバック値が使用されます。

## 4. 未実装だが今後重要なもの

将来の本格パッケージングおよび「MANA」ブランドへの統合に向けた拡張余地です。

*   **Voice/TTS (音声出力の実装化)**
    *   `docs/voice_accessibility_plan.md` の設計方針（早口厳禁、ウィスパーに近い穏やかなトーン）は明記されているものの、実際の音声合成エンジン（Style-BERT-VITS2のNoaモデルや、irodori_TTSなど）を用いた音声出力・再生の実装は行われていません。
*   **Accessibility の更なる強化**
    *   音声認識（Speech-to-Text）によるハンズフリー入力や、画面読み上げ（スクリーンリーダー）に完全に適応したHTMLセマンティクスの最適化。
*   **PersonaGuard の拡張**
    *   現在定義されているのは一部の主要な言葉（必ず、病気、治療など）の置換であるため、シチュエーションに応じた心理的安全のための禁止表現辞書の充実。
*   **EmotionEngine による「時間的低圧制御」**
    *   ストレス状態に応じて、AIの返答文字が現れるスピード（アニメーション速度）を遅くしたり、返答までに少し「間（ウェイト）」を持たせるなど、時間軸でのプレッシャーコントロール。
*   **カメラ連携機能の実装化**
    *   `docs/camera_future_plan.md` に記載されている、カメラからの表情分析（FACS：表情アクションユニットの検出）や視線、心拍数推定を用いた「視覚的な能動チェックイン」の実装。
*   **マルチモーダル（MANA）連携**
    *   Live2Dなどのキャラクターアバターと音声、状態推定を連動させ、単なるUI画面ではなく、一人の「寄り添うコンパニオン」としての統合。

## 6. 今後の優先順位

### 今すぐやるべき（デモ・展示・ピッチ向け）
1.  **実演用のシナリオスクリプト、ピッチ動画用のスライド・構成の作成**
    *   すでに実装済みの2画面比較デモと、アクセシビリティタブでの動作の流れを最も効果的に見せるプレゼンフローの定義。
2.  **プロフィール保存処理の保護**
    *   `personal_profiles.json` の書き込み時に一時ファイルを作成してリネームする安全なセーブ処理に変更し、ファイル破損リスクを排除する。

### 後回しでいい（中長期的な本製品化向け）
1.  **音声合成 (TTS) やカメラ画像処理の本格実装**
    *   ハードウェアや環境依存が多く、デモ展示時のトラブル要因になりやすいため、まずはモック/仕様書（設計書）での説明に留めるのが無難です。
2.  **Gemini API の本格接続とプロンプトの過度なチューニング**
    *   現状のローカルフォールバックでデモ機能として高い完成度に達しているため、デモ用途に限定するならばAPIキー接続の最適化に時間を割く優先度は低いです。
3.  **オンラインデプロイ（Cloud Run等）**
    *   ローカルでの実行、あるいはデモ動画のキャプチャがあれば十分プレゼンテーションが可能なため、オンライン対応は後回しで構いません。
